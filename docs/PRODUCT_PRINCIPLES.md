# LUMEN — Principios de producto

Este documento define una primera base de principios de producto para LUMEN. No reemplaza al Figma oficial ni define identidad visual final. Su objetivo es ayudar a tomar decisiones coherentes sobre experiencia, tono, jerarquia y claridad mientras el proyecto evoluciona.

## Proposito

LUMEN es una plataforma digital para acompanar procesos de bienestar, aprendizaje y atencion profesional.

Debe funcionar como un espacio confiable donde una persona pueda:

- Conocer la propuesta de LUMEN.
- Reservar consultas profesionales.
- Acceder a cursos asincronicos.
- Explorar recursos fisicos y digitales.
- Gestionar su actividad dentro de un area privada.

El producto debe transmitir que detras de la plataforma hay una institucion seria, ordenada y responsable, pero sin sentirse fria, distante o burocratica.

## Que debe sentir un usuario en los primeros segundos

En los primeros segundos, una persona deberia sentir:

- Confianza: entiende rapidamente que esta en un espacio profesional.
- Claridad: sabe que puede hacer y hacia donde avanzar.
- Calma: la interfaz no la abruma ni le exige entender demasiadas cosas juntas.
- Cercania: el lenguaje se siente humano, simple y respetuoso.
- Seguridad: los procesos importantes, como reservar, pagar, registrarse o cargar datos personales, se perciben cuidados.

La primera impresion debe estar mas cerca de una institucion seria de salud o educacion que de una startup experimental.

## Principios de UX

### 1. Claridad antes que decoracion

Cada pantalla debe dejar claro:

- Donde esta la persona.
- Que puede hacer ahi.
- Cual es el siguiente paso recomendado.
- Que informacion necesita para decidir.

La experiencia no debe depender de que el usuario explore demasiado para entenderla.

### 2. Flujos simples y directos

Las acciones principales deben sentirse simples, como en plataformas de uso cotidiano:

- Elegir.
- Revisar.
- Confirmar.
- Continuar.

Reservar una consulta, explorar un curso o solicitar un producto no deberia sentirse como completar un tramite complejo.

### 3. Menos carga mental

El usuario no debe tener que recordar informacion de una pantalla a otra. Cuando una accion tenga varios pasos, la plataforma debe mostrar contexto suficiente para que la persona entienda que esta haciendo.

Ejemplos:

- Si reserva una consulta, debe ver profesional, fecha, horario y modalidad antes de confirmar.
- Si compra o solicita algo, debe ver que esta eligiendo y que pasara despues.
- Si entra a su cuenta, debe ver rapidamente sus turnos, cursos, recursos y mensajes relevantes.

### 4. Estados vacios utiles

Un estado vacio no debe sentirse como un error. Debe explicar brevemente que falta y ofrecer un camino claro.

Ejemplo de intencion:

- Si no hay cursos habilitados, ofrecer explorar cursos.
- Si no hay turnos, ofrecer reservar una consulta.
- Si no hay mensajes, explicar que ahi apareceran novedades importantes.

### 5. Acciones importantes siempre visibles

Las acciones principales deben ser faciles de encontrar:

- Reservar consulta.
- Explorar cursos.
- Ver catalogo.
- Iniciar sesion.
- Ir a Mi Espacio.
- Contactar a LUMEN.

No deben quedar escondidas dentro de textos largos o menus poco evidentes.

### 6. Mobile como experiencia real

La plataforma debe funcionar bien en pantallas chicas. No alcanza con que "se vea"; debe poder usarse comodamente.

Hay que evitar:

- Textos comprimidos.
- Tarjetas demasiado angostas.
- Botones dificiles de tocar.
- Menus abiertos permanentemente.
- Informacion clave fuera de pantalla.

## Principios de producto

### 1. LUMEN debe ordenar la experiencia

La plataforma debe ayudar a ordenar tres grandes caminos:

- Consultas profesionales.
- Cursos.
- Catalogo de recursos.

Aunque el proyecto crezca, el usuario siempre debe poder entender rapidamente cual de esos caminos necesita.

### 2. La confianza se construye con transparencia

Antes de que una persona reserve, compre o cargue datos, debe entender:

- Que esta solicitando.
- Quien lo acompana.
- Que pasa despues.
- Cuales son los limites actuales del proceso.

Si una funcion todavia esta en preparacion, no debe prometer mas de lo que realmente hace.

### 3. El area privada debe sentirse como un espacio propio

"Mi Cuenta" o "Mi Espacio" debe ser el lugar donde el usuario entiende su relacion con LUMEN:

- Sus turnos.
- Sus cursos.
- Sus recursos.
- Sus pedidos.
- Sus mensajes.
- Sus datos personales.
- Su facturacion.

Debe sentirse como una plataforma organizada, no como una pagina suelta.

### 4. El panel admin debe priorizar control y orden

El admin no necesita ser decorativo. Necesita ser claro, predecible y seguro.

Debe ayudar a gestionar:

- Usuarios.
- Cursos.
- Especialistas.
- Turnos.
- Catalogo.
- Mensajes.
- Facturacion.
- Contenido.

Las acciones destructivas o sensibles deben pedir confirmacion y explicar que va a pasar.

### 5. El panel especialista debe priorizar agenda y seguimiento

El especialista necesita ver con claridad:

- Sus turnos.
- Sus pacientes.
- El historial de consultas.
- Notas privadas.
- Datos relevantes para seguimiento.

La experiencia debe sentirse profesional y reservada, no social ni informal.

### 6. Crecer sin perder simplicidad

LUMEN puede sumar nuevas funciones en el futuro, pero cada incorporacion debe responder una pregunta:

¿Esto ayuda al usuario a avanzar con mas claridad, confianza o autonomia?

Si la respuesta no es clara, conviene dejarlo como pendiente antes de sobrecargar la plataforma.

## Tono de comunicacion

El tono de LUMEN debe ser:

- Claro.
- Profesional.
- Cercano.
- Sereno.
- Respetuoso.
- Directo.

Debe evitar:

- Frases demasiado tecnicas.
- Mensajes frios o impersonales.
- Promesas exageradas.
- Lenguaje de marketing vacio.
- Textos largos cuando una indicacion simple alcanza.

### Como deberian sonar los textos

Los textos deberian ayudar sin invadir.

Ejemplos de intencion:

- "Elegí un horario disponible para continuar."
- "Tu consulta quedó reservada correctamente."
- "Cuando tengas cursos habilitados, van a aparecer acá."
- "Si necesitás ayuda, podés escribirnos desde Contacto."

### Mensajes de error

Los errores deben:

- Explicar que paso.
- Decir que puede hacer el usuario.
- Evitar culpar a la persona.
- Evitar mostrar errores tecnicos crudos cuando sea posible.

### Mensajes de exito

Los mensajes de exito deben:

- Confirmar la accion.
- Indicar el siguiente paso si corresponde.
- Ser breves.

## Que NO debe ser LUMEN

LUMEN no debe sentirse como:

- Una startup ruidosa o experimental.
- Una landing llena de promesas genericas.
- Una plataforma fria y burocratica.
- Un marketplace desordenado.
- Un panel tecnico dificil de entender.
- Una app que obliga al usuario a adivinar el siguiente paso.
- Un sitio que mezcla funciones sin jerarquia.

Tampoco debe prometer funciones que todavia no estan listas. Si algo esta preparado para una etapa futura, debe comunicarse con cuidado o mantenerse fuera de la experiencia principal.

## Reglas para futuras decisiones de diseno

### 1. Priorizar confianza sobre impacto visual

Si una decision visual llama mucho la atencion pero reduce claridad, debe revisarse.

### 2. Priorizar caminos principales

La navegacion debe mantener visibles los caminos centrales:

- Consultas profesionales.
- Cursos.
- Catalogo.
- Contacto.
- Mi Espacio, cuando el usuario esta logueado.

### 3. No agregar complejidad sin necesidad

Antes de sumar una nueva seccion, boton o mensaje, preguntarse:

- ¿Ayuda al usuario?
- ¿Reduce dudas?
- ¿Evita un error?
- ¿Hace mas claro el siguiente paso?

Si no cumple al menos una de esas funciones, probablemente puede esperar.

### 4. Cada pantalla debe tener una accion principal

Una pantalla no deberia competir consigo misma. Debe quedar claro cual es la accion mas importante en ese contexto.

### 5. Los placeholders deben revisarse antes de publicar

Todo texto provisorio, generico o tecnico debe pasar por revision antes de considerarse final.

Esto incluye:

- Estados vacios.
- Mensajes de error.
- Mensajes de exito.
- Botones.
- Titulos.
- Formularios.
- Textos legales.
- Textos del admin visibles para usuarios internos.

### 6. Identidad visual: paleta cromatica ya definida, resto pendiente del Figma

La paleta cromatica oficial de LUMEN ya esta definida y puede integrarse desde el codigo (tokens de color en `app/globals.css`).

Siguen pendientes del Figma definitivo, y no deben resolverse desde el codigo hasta recibirlo:

- Logo.
- Isotipo.
- Tipografia.
- Iconografia.
- Espaciados.
- Radios.
- Sombras.
- Componentes finales.
- Layouts visuales definitivos.

Este documento no define esos elementos ni un sistema visual definitivo mas alla de la paleta cromatica.

### 7. Mantener coherencia entre secciones

Las mismas acciones deben llamarse igual en toda la plataforma.

Ejemplos a revisar:

- "Mi Cuenta" vs. "Mi Espacio".
- "Turnos" vs. "Consultas profesionales".
- "Catalogo" vs. "Recursos".
- "Mensajes" vs. "Notificaciones".

La coherencia de nombres ayuda a que la plataforma se sienta mas seria y mas facil de usar.
